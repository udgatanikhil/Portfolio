# Vault — Multi-Role Project Delivery Cockpit & AI Assistant

*Published on July 21, 2026 • 12 min read • Category: Architecture & AI Ops • By Nikhil Udgata*

Vault is an enterprise-grade, multi-role project management cockpit designed for modern engineering teams. It bridges the gap between client visibility, engineering execution, and program management by uniting structured intake, resource capacity planning, and bi-directional Jira Cloud syncing into a single, real-time workspace.

---

## 🗺️ The Delivery Journey & System Architecture

Vault aligns four organizational roles into a continuous delivery loop, synchronized in real-time via a Redis Pub/Sub SSE tunnel and an incoming Jira webhook stream:

```
[1. Client Intake & RICE] ──> [2. TPM Capacity Allocation] ──> [3. AE Execution & Standup] ──> [4. Admin Cockpit & AI Analytics]
         ▲                                                                                                    │
         └────────────────────────── Bi-Directional Jira & Webhook Loop ─────────────────────────────────────┘
```

![Vault E2E System Architecture & Delivery Journey](/vault_architecture_diagram.jpg)
*Figure 1: High-level architectural flowchart of Vault showing the real-time event pipeline and Jira integration.*

### Technical Stack Breakdown:
* **Frontend**: React SPA, Vite, Tailwind CSS (V3), custom SVG Gantt charting engine, real-time Server-Sent Events (SSE) hooks.
* **Backend**: Node.js, Express, cursor-paginated Jira REST API integrations, Redis pub/sub broker, PostgreSQL database with dynamic SQL translation.
* **AI Orchestration**: Dify Docker stack (wrapping Ollama/Llama 3), implementing the lean-prompt pattern to minimize token usage and cache summaries in Redis (4h TTL).

## 🛠️ Local Installation & Quick Start

### Prerequisites
* Node.js (v20+)
* PostgreSQL (v14+)
* Redis Server (v6+)
* Docker (for Dify AI stack)

### 1. Clone & Install Dependencies
```bash
# Install backend dependencies
cd backend
npm install

# Install frontend dependencies
cd ../frontend
npm install
```

### 2. Configure Environment Variables
Create a `.env` file in the `backend/` directory:

```env
PORT=3001
DB_HOST=127.0.0.1
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=your_password
DB_NAME=playground
REDIS_URL=redis://127.0.0.1:6379

# Jira Integration
JIRA_BASE_URL=https://your-domain.atlassian.net
JIRA_EMAIL=your-email@domain.com
JIRA_API_TOKEN=your_atlassian_api_token
JIRA_WEBHOOK_SECRET=your_webhook_verification_secret

# Dify AI
DIFY_BASE_URL=http://localhost/v1
DIFY_API_KEY=your_dify_api_key
```

### 3. Database Initialization & Migrations
Vault uses PostgreSQL. To seed schemas and load default migrations:

```bash
cd backend
npm run db:init
npm run db:migrate
npm run db:seed
```

### 4. Running the Application
```bash
# Start backend service (Express on port 3001)
cd backend
npm run dev

# Start frontend dev server (Vite on port 5173 with HMR)
cd ../frontend
npm run dev
```

## ⚡ Key Synchronization Engines

### Server-Sent Events (SSE) Sync
Vault coordinates browser states without page refreshes. When a TPM assigns a ticket:
1. The backend writes the allocation and publishes an event to the `pm:events` Redis channel.
2. The Express SSE server catches the Redis event and pushes it to connected browsers.
3. The frontend `usePmEvents()` hook intercepts the payload, debounces for 300ms, and reloads the affected component state.

### Bi-directional Jira Webhook Loop
1. When a ticket status changes in Jira, a webhook is fired to `POST /api/jira/webhook`.
2. Vault records the transition, alerts the assigned engineer's Inbox, and prompts them for an audit comment.
3. The engineer's explanation is posted directly back to Jira as an issue comment via `jira.addComment()`.
